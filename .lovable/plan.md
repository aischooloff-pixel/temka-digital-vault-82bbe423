

# Полный продакшн-аудит и план исправлений

## Найденные критические баги

### 1. Захардкоженная ссылка на поддержку
- **Файлы**: `ProductDetails.tsx:175`, `OrderFailed.tsx:26` — ссылка `https://t.me/paveldurov` вместо реального саппорта
- **Решение**: Использовать хук `useSupportUsername()` как в остальных страницах

### 2. Статус `expired` отсутствует в типах
- `check-payment/index.ts` ставит `payment_status: "expired"`, но в `types/database.ts` этого значения нет
- В UI история заказов не показывает лейбл для истёкших платежей
- **Решение**: Добавить `'expired'` в тип и в `PAYMENT_STATUS_LABELS`

### 3. Промокод расходуется при создании инвойса, а не при оплате
- `create-invoice/index.ts:112-125` — `used_count` увеличивается сразу при создании заказа
- Если пользователь не оплатил — промокод "сгорел" впустую
- **Решение**: Убрать инкремент из `create-invoice`, перенести в `cryptobot-webhook` и `check-payment` (после подтверждения оплаты)

### 4. Нет валидации количества товара в корзине
- Можно увеличить количество больше `stock` — сервер вернёт ошибку, плохой UX
- **Решение**: Ограничить `updateQuantity` до `Math.min(qty, product.stock)`, заблокировать кнопку "+"

## Проблемы безопасности

### SEC-1: initData не валидируется в платёжных функциях (Средний)
- `telegram-auth` существует, но не вызывается. Платёжные функции принимают `telegramUserId` от клиента без проверки
- **Решение**: Передавать `initData` в `create-invoice` и `pay-with-balance`, валидировать на сервере

### SEC-2: Race condition при списании баланса (Низкий)
- И `cryptobot-webhook`, и `check-payment` могут обработать один платёж
- Есть проверка `payment_status !== 'paid'`, но теоретически возможно двойное списание
- **Решение**: Проверять количество обновлённых строк после UPDATE

### SEC-3: Нет rate limiting (Низкий)
- Edge functions не ограничивают частоту запросов
- **Рекомендация**: Добавить после запуска

### Что уже хорошо реализовано:
- ✓ Подпись вебхука CryptoBot (HMAC-SHA-256)
- ✓ Проверка админ-прав в боте (ADMIN_TELEGRAM_IDS + admin_users)
- ✓ Параметризованные запросы через Supabase SDK (нет SQL injection)
- ✓ React авто-экранирование (нет XSS)
- ✓ RLS политики на таблицах

## План реализации

### Задача 1: Исправить захардкоженные ссылки поддержки
- `ProductDetails.tsx` — импортировать `useSupportUsername`, заменить ссылку
- `OrderFailed.tsx` — то же самое

### Задача 2: Добавить статус `expired`
- `types/database.ts` — добавить `'expired'` в тип `payment_status` и лейбл `'Истёк'`
- `Account.tsx` — добавить иконку и цвет для `expired`

### Задача 3: Перенести инкремент промокода на момент оплаты
- Удалить блок инкремента из `create-invoice/index.ts`
- Добавить инкремент в `cryptobot-webhook/index.ts` (в `handleOrderPayment`)
- Добавить инкремент в `check-payment/index.ts` (в блоке обработки оплаты)

### Задача 4: Валидация количества в корзине
- `Cart.tsx` — заблокировать кнопку "+" при `quantity >= stock`
- Показать предупреждение при достижении лимита

### Задача 5: Защита от двойного списания баланса
- В `cryptobot-webhook` и `check-payment` — проверять `.match({ payment_status: 'awaiting' })` при UPDATE, чтобы обновлять только если статус ещё не `paid`

## Чеклист перед запуском

1. Очистить тестовые товары и отзывы из БД
2. Установить `WEBAPP_URL` на продакшн URL
3. Установить секрет `BOT_USERNAME`
4. Установить `support_username` в `shop_settings`
5. Установить продакшн `ADMIN_TELEGRAM_IDS`
6. Перерегистрировать вебхук: `GET /functions/v1/telegram-bot?setup=true`

## Вердикт

**Проект близок к продакшн-готовности.** Архитектура крепкая, оплата с идемпотентностью, админка полная, данные динамические.

**Обязательно исправить** (задачи 1–4): хардкод поддержки, тип expired, промокод, валидация корзины.

**Желательно** (задача 5): защита от race condition.

**Рекомендовано**: валидация initData в платёжных функциях.
